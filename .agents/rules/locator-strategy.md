# Locator Selection Strategy (Applicable to All Automation Frameworks)

> Locator stability and readability determine the long-term health of an automation framework.
>
> Core principle: **Never locate elements based on DOM structure tied to styling.**
>
> Always prefer locators based on semantic, accessible, and stable attributes.

---

# 1. Master Priority Map

Choose locators using the following priority order (highest to lowest):

| Priority | Locator Type                                                      |
| -------- | ----------------------------------------------------------------- |
| 1        | Accessibility / ARIA attributes                                   |
| 2        | Dedicated test attributes (`data-testid`, `data-test`, `data-qa`) |
| 3        | Primary identifiers (`id`, `resource-id`, `name`)                 |
| 4        | Framework semantic locator APIs (`getByRole`, `getByLabel`, etc.) |
| 5        | CSS Selectors                                                     |
| 6        | XPath (last resort only)                                          |

---

# 2. Stability Rules

Every locator must satisfy the following requirements:

* Match exactly one target element within the intended scope.
* Remain stable across UI layout changes.
* Survive DOM restructuring such as:

  * Additional wrapper elements
  * Nested containers
  * Flexbox/Grid refactoring
  * Styling changes

## Strictly Prohibited

Do not use:

### Dynamic CSS Classes

```text
css-1n2xyz-btn
sc-abc123
chakra-button-xyz
```

### Positional Selectors

```css
div:nth-child(3)
li:nth-of-type(2)
```

unless no better option exists.

### Auto-generated IDs

```text
react-123
mui-456
input-789
```

generated by frameworks and subject to change.

### Absolute XPath

```xpath
/html/body/div[3]/div[2]/form/button
```

or

```xpath
//div[3]/div[2]/form/button
```

These are highly fragile and must not be used.

---

# 3. Locator Validation Process

Before adding any locator to the automation codebase, verify:

## Validation Checklist

### 1. Uniqueness

Does the locator match exactly one element?

### 2. Interactability

Is the matched element the actual user-interactable element?

Avoid:

* Hidden elements
* Overlay elements
* Placeholder containers
* Non-clickable wrappers

### 3. Persistence

After page refresh or navigation, does the locator still work?

### 4. State Stability

Test the locator across different application states:

* Loading state
* Loaded state
* Empty state
* Populated state
* Error state
* Disabled state

The locator should remain reliable whenever possible.

---

# 4. Locator Design Principles

Prefer user-facing semantics over implementation details.

Good examples:

```text
Submit button
Email field
Password field
Shopping cart badge
Delete customer button
```

Bad examples:

```text
div > div > button
.btn-primary
.card:nth-child(2)
```

---

# 5. Preferred Locator Examples

## Accessibility-Based

```typescript
page.getByRole('button', { name: 'Login' })

page.getByRole('textbox', { name: 'Email' })

page.getByRole('checkbox', { name: 'Remember me' })
```

## Label-Based

```typescript
page.getByLabel('Email')

page.getByLabel('Password')
```

## Test Attribute-Based

```typescript
page.getByTestId('login-button')

page.getByTestId('email-input')
```

## Stable ID-Based

```typescript
page.locator('#loginButton')

page.locator('[name="email"]')
```

---

# 6. Fallback Strategy

Use the strongest available locator.

Recommended decision flow:

```text
Accessibility / ARIA
        ↓
Dedicated Test Attributes
        ↓
Stable ID / Name
        ↓
Framework Semantic Locators
        ↓
CSS Selector
        ↓
XPath (Last Resort)
```

If XPath is required:

* Document the reason.
* Use the shortest and most stable XPath possible.
* Avoid positional indexing whenever possible.

# 8. Review Checklist

Before merging any automation code:

* Locator matches exactly one element.
* Locator is semantic and user-focused.
* Locator is not dependent on styling.
* Locator is not dependent on DOM hierarchy.
* Locator survives page refresh.
* Locator survives common UI refactors.
* No dynamic classes are used.
* No positional selectors are used unless justified.
* XPath is used only as a last resort.
* The most stable locator available has been selected.
